## 2. 当前批量补齐前提判断

### 2.1 当前是否适合做全量批量推进

**结论**：**不适合**“一把梭把所有剩余页面一次性改完”。（已确认）

- 剩余页面类型不一：标准 CRUD、复杂关联（用户/角色）、日志只读+删除、字典 type/data 双实体、接口管理仅查改，若同一节奏推进易出现接口口径不统一、公共链路未稳定导致多模块返工。
- 用户/角色与菜单/权限强相关，且用户 Update 为 PUT 无 :id、角色依赖 roleMenuTreeselect/roleDeptTreeselect 树结构，需先联调确认再作为后续参考。
- 认证路径（刷新/登出）**已确认**：代码中 auth.ts 已使用 /v1/refresh_token、/v1/logout，与后端一致；联调行为建议验证。

### 2.2 当前最适合采用的推进模式

| 模式 | 说明 |
|------|------|
| **一个总 Plan + 多个子 Plan** | 以本文档为总方案，按批次拆子 Plan（如“Batch 0 公共链路”“Batch 1 用户管理”“Batch 2 角色管理”“Batch 3 日志+字典+接口管理”），每子 Plan 绑定明确页面与母版。 |
| **按页面类型批量推进** | 先标准 CRUD 页（可高复用母版），再复杂关联页（用户/角色），再日志/辅助页；同类页面共享同一套完成定义与验收口径。 |
| **先沉淀母版，再批量复制** | 岗位、参数配置、部门已为母版；用户/角色先做一轮并联调通过后，可视为“带关联的 CRUD 母版”，再复用到字典等。 |
| **公共链路优先** | Batch 0 先确认或修正认证路径与公共链路，再开 Batch 1～3，避免多条子 Plan 同时改 auth/request。 |

## 3. 页面类型分层

以下仅针对**剩余待补齐页面**分类；已完成页面（登录、菜单、部门、参数配置、岗位、Dashboard 等）不重复列出。

### A 类：标准 CRUD 列表页

| 项 | 内容 |
|----|------|
| **类型定义** | 分页表 + 搜索区 + 新增/编辑/删除弹窗，无树、无复杂关联下拉。 |
| **典型特征** | 列表 GET 分页、详情 GET/:id、提交 POST/PUT、删除 DELETE body ids。 |
| **当前包含** | **字典类型**（sys-dict-type）、**字典数据**（sys-dict-data）、**接口管理**（sys-api 仅列表+编辑）已接入；字典数据页存在引用残留需修正。 |
| **适合复用的母版** | 岗位管理、参数配置。 |
| **推荐批量推进程度** | 已交付；下一批为联调验证 + 字典数据引用修正。 |
| **主要风险** | 无；dict 已补充 getDictTypeOptionSelect、DictTypeOption 导出。 |
| **验收重点** | 列表/分页/搜索、新增/编辑/删除（若有）、成功失败提示、刷新。 |

### B 类：树形管理页

| 项 | 内容 |

### D 类：日志 / 记录型页面

| 项 | 内容 |

## 6. 母版复用策略

### 6.1 当前可作为母版的页面

| 母版 | 类型 | 落点 | 适用对象 |
|------|------|------|----------|
| **岗位管理** | 标准分页 CRUD | `views/admin/sys-post/index.vue`、`api/core/post.ts` | 角色、字典、日志、接口管理的列表与弹窗结构 |
| **参数配置** | 标准分页 CRUD | `views/admin/sys-config/index.vue`、`api/core/config.ts` | 同上；分页、搜索、弹窗、删除二次确认、requestClient 用法 |
| **用户管理** | 复杂关联 CRUD | `views/admin/sys-user/index.vue`、`api/core/user.ts` | 带部门/角色/岗位下拉的 CRUD 页；角色管理可参考 |
| **部门管理** | 树形管理 | `views/admin/sys-dept/index.vue`、`api/core/dept.ts` | 任何树形维护页；编辑时父级排除自身及子节点（filterDeptTreeExcludeNode） |
| **菜单管理** | 特殊树形+关联 | `views/admin/sys-menu/index.vue` | 仅作约束参考（component/icon/access）；不复制为通用母版 |

### 6.2 哪一类页面优先参考哪个母版

- **用户管理**：已实现，可作为**复杂关联页母版**；部门/角色/岗位下拉已接 getDeptTreeApi、getRolePage、getPostPage。
- **角色管理**：列表与弹窗骨架优先参考**用户管理**或**岗位/参数配置**；角色表单项内菜单树/部门树参考**菜单管理**的父级树与**部门**的 getDeptTreeApi，以及 roleMenuTreeselect、roleDeptTreeselect。
- **字典类型/数据**：类型列表、数据列表（或 Tab）参考**参数配置**的分页表与弹窗；若类型与数据分两个子表，可先做类型 CRUD，再做数据按 type 过滤的 CRUD。
- **登录日志、操作日志**：仅列表+筛选+删除，参考**参数配置**的列表与分页与删除二次确认，不做新增/编辑弹窗。
- **接口管理**：列表+编辑，参考**参数配置**的列表与编辑弹窗；不提供新增与删除入口。

### 6.3 母版复用时保持一致与可适配部分

| 保持一致 | 允许局部适配 |
|----------|----------------|
| requestClient、code/data 解包、分页参数 pageIndex/pageSize | 接口路径、请求体字段、表格列、表单字段 |
| 删除二次确认、列表刷新、成功/失败 message | 搜索项、筛选项、是否展示新增/编辑/删除 |
| 弹窗打开/关闭、详情回填再提交 | 树组件、下拉数据源（部门树、角色列表等） |
| 表格 loading、空态、错误态处理 | 字典 type/data 双表或 Tab 布局 |

### 6.4 不适合硬套母版的页面

- **角色管理**：菜单树/部门树为多选或勾选，与部门“单选父级”不同，需单独适配 roleMenuTreeselect、roleDeptTreeselect 返回结构。
- **字典**：type 与 data 双实体，可能需 Tab 或主从表，与单表 CRUD 略有差异。
- **接口管理**：仅查改，无新增/删除，按钮与权限展示需与母版区分。

### 6.5 建议先沉淀再复用的模式

- **用户管理**：先做一轮并联调通过后，可沉淀“带部门/角色/岗位下拉的 CRUD 页”模式，供后续类似表单复用。
- **角色管理**：菜单/部门树选择联调通过后，可沉淀“树选择与提交结构”，供其他需树多选的配置页参考。

## 7. 并行开发建议

### 7.1 哪些批次适合并行

- **Batch 0** 与 **Batch 1**：可并行（Batch 0 动 auth.ts，Batch 1 动 user.ts + sys-user 页），但若 Batch 0 结论为“需改 auth”，建议先合入再开 Batch 1，避免 request 行为不一致。
- **Batch 1** 与 **Batch 3**：可并行（用户管理 vs 日志/字典/接口管理），不同文件、无交叉依赖。
- **Batch 2** 与 **Batch 3**：可并行（角色 vs 日志/字典/接口管理）。
- **Batch 1** 与 **Batch 2**：建议串行（先用户后角色），便于复用同一分页表模式与联调经验；若资源充足可并行，但需注意两处同时依赖“角色列表”“部门树”等接口时的联调顺序。

### 7.2 哪些页面适合并行

- Batch 3 内部：登录日志、操作日志、字典、接口管理可拆为 2～4 个子 Plan 并行（不同 views 与 api 文件）。
- 用户管理页与角色管理页：不同目录（sys-user、sys-role），可并行开发，但联调建议用户先、角色后。

### 7.3 哪些页面不应并行

- 不宜多个子 Plan 同时改 `api/core/auth.ts`、`api/request.ts`、`router/access.ts`。
- 不宜在未确认认证路径前，多条线同时加依赖 401/refresh 的新页面。

### 7.4 容易冲突的文件层级

- `api/core/*.ts`：若多批次同时新增或修改同一文件（如 user.ts 既扩 getinfo 又扩 sys-user CRUD），需约定归属或分批合入。
- `router/access.ts`：不在此次批量补齐范围内，仅不修改即可。
- 新增 `views/admin/*`：不同模块不同目录，冲突概率低。

### 7.5 推荐的主线 / 支线搭配

- **主线**：Batch 0～3 前端均已交付；当前推荐主线为**联调验证**（公共链路 → 用户/角色 → 参数配置/字典/日志/接口管理）。字典数据页引用已通过 dict 补充导出修正。
- **支线**：无新页面开发；若需扩展业务模块，可复用现有母版与 api/core 模式。

## 8. 每一批次的完成定义

### Batch 0

- 刷新/登出路径与后端或代理一致（或结论为无需修改）；登录→getinfo→menurole 全流程无回归；登录后能进首页且菜单正确。

### Batch 1（用户管理）

- 前端已实现：列表、分页、搜索、新增/编辑/删除弹窗、部门/角色/岗位下拉；编辑为 PUT 无 :id、body 带 userId。
- 验收口径：与后端 GET/POST/PUT/DELETE /v1/sys-user 联调通过；分页参数、Update 路径与 body、getinfo 与列表字段区分清晰。

### Batch 2（角色管理）

- 列表：分页、搜索与后端 GET /v1/role 一致。
- 新增/编辑：弹窗含菜单树、部门树（roleMenuTreeselect、roleDeptTreeselect）选择；提交后角色与菜单/部门关联生效（如 menurole 变化）。
- 删除：二次确认、列表刷新。
- 联调：树接口返回结构与前端树组件兼容。

### Batch 3（日志+字典+接口管理）

- **登录/操作日志**：列表加载、筛选、分页、删除（二次确认）；与后端 GET/DELETE 一致。
- **字典**：类型列表与数据列表（或 Tab）可查可维护；增删改与后端 dict/type、dict/data 一致。
- **接口管理**：列表可查、编辑可保存；与后端 GET/PUT sys-api 一致；无新增/删除。
- 各模块成功/失败提示、刷新、空态与错误态。

### Batch 4

- 全链路联调通过；参数配置与后端联调验证（若尚未做）；权限与异常场景验收；文档与已知问题收敛（见 PRD 与实施计划验收节）。

## 9. 风险、阻塞项与前置确认

### 9.1 当前阻碍批量推进的关键问题

- **认证路径**：已确认代码中 auth.ts 使用 /v1/refresh_token、/v1/logout，与后端一致；联调行为建议验证。
- **用户 Update**：PUT 无 :id、body 带 userId，若前端按 :id 写会失败；需与实施计划、后端 router 一致。（已确认 router 为 PUT ""）
- **角色树接口**：roleMenuTreeselect、roleDeptTreeselect 返回结构与前端树组件是否兼容。（待确认）

### 9.2 必须先确认的项

- 数据权限 enabledp 是否启用及对用户/角色页面的影响（若启用则可能影响列表与操作范围）。刷新/登出路径代码已与后端对齐。

### 9.3 可边做边确认的项

- 字典 type/data 的接口路径与分页、筛选参数。
- 接口管理是否仅列表+编辑（无新增/删除）。
- 参数配置、用户管理真实环境联调（代码口径已核对一致）。

### 9.4 会因公共链路不稳定而返工的模块

- 所有依赖 request 解包、401 处理、Bearer 的页面，在认证路径错误时可能集体返工。
- 用户/角色若分页参数与后端不统一，需统一或适配后再推广到字典/日志。

### 9.5 暂不建议批量启动的模块

- 验证码接入（登录当前固定 code/uuid）。
- 大规模重构 access、request、getinfo 映射。
- 数据权限在前端的大幅改造（enabledp 未确认前）。

### 9.6 最容易出现“表面全做完、质量不一”的点

- 仅建页面骨架未接真实接口或未联调。
- 删除接口入参（body ids）与后端不一致。
- 分页参数（pageIndex/pageSize）与后端 dto 不一致。
- 角色树选择提交结构与后端期望不一致。

## 11. 后续拆分建议

### 11.1 如何基于本文档继续拆单轮开发任务

- 从**第 4 节**取批次目标与包含模块；从**第 5 节**取页面与落点、母版、依赖联调程度。
- 单轮任务聚焦一个批次或一个页面（如“本轮只做用户管理”），从**第 6 节**取母版与复用策略，从**第 8 节**取该批完成定义。
- 联调与验收从**第 9 节**与 PRD/实施计划对应章节执行。

### 11.2 如何从批次拆到单模块/单页面

- **Batch 1**：可拆为“用户 API 封装”+“用户管理页骨架与列表”+“用户新增/编辑弹窗”+“用户删除”（与实施计划 T-006～T-010 对应）。
- **Batch 3**：可拆为“登录日志”“操作日志”“字典类型/数据”“接口管理”四个子 Plan，再按需拆为 API + 页面。
- 每子 Plan 绑定明确落点文件（见第 5 节备注列），避免范围蔓延。

### 11.3 如何避免拆着拆着失控

- 严格以 PRD 与实施计划为需求与任务基线；本文档只做批次与母版与并行策略，不替代单任务完成定义。
- 每批完成后做该批验收再开下一批；避免多批同时“进行中”且无联调结论。
- 不修改 access、request、getinfo 映射等已结案链路；新增菜单 component 必须在 validViewPathSet 内。

### 11.4 在保持总方案一致的前提下逐轮落代码

- 按 Batch 0 → 1 → 2 → 3 → 4 顺序推进，或按第 10 节子 Plan A/B 执行（先公共链路校验，再联调验证）。
- 每轮交付物：对应批次页面+API+联调结论（或待办）；更新实施计划中任务状态（如 T-006～T-012、T-013～T-017）。
- 总方案（本文档）在批次范围与母版策略未变时可不频繁改；仅当剩余页面或后端能力有较大变化时再同步更新本文档。
