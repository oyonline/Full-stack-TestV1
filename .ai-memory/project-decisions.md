# 项目关键决策

## 系统设置

- `参数设置` 是唯一正式系统设置入口。
- 右上角偏好设置已移除，不再作为独立入口保留。
- 界面偏好被定义为全局系统设置，而不是个人偏好。
- 运行时系统设置以后端配置为真相源，不以后端默认值或浏览器本地缓存为主。

## 权限模型

- 用户支持多角色。
- 菜单按主角色生效。
- 按钮权限按所有角色并集生效。
- 接口权限当前按主角色 key 走 Casbin 校验，不是按多角色并集校验。
- 本期不做运行时角色切换。
- 数据权限继续关闭，不纳入当前模型。

## 菜单与路由

- `sys_menu` 是菜单真相源。
- 前端动态路由只负责正式映射，不再长期维护旧路径兼容。
- 父级分组菜单统一使用 `RouteView`，不再使用会渲染二次侧栏的 `BasicLayout`。
- `IFrameView` 用于承接后端静态页面，如表单构建。

## 字典管理

- 字典管理的正式主链路固定为：
  - `字典类型目录页`
  - `字典类型详情页`
- 当前类型下的字典数据维护统一收口到类型详情页，不再保留独立的“全部字典数据”产品页。
- `/admin/sys-dict-data` 只保留一版隐藏 redirect 用于吸收历史书签，不再作为正式页面、菜单路径或数据库正式值继续维护。
- 当前类型下的数据读取权限并入字典类型查看权限，不再保留独立的 `admin:sysDictData:list` / `admin:sysDictData:query` 产品权限概念。
- `admin:sysDictData:add` / `admin:sysDictData:edit` / `admin:sysDictData:remove` 继续保留，供类型详情页里的数据操作使用。
- 菜单种子与迁移层的正式收口版本为 `1775300000000_remove_dict_data_page.go`；后续不要再手工恢复 `menu_id=59` 或 `menu_id=240`。

## 后台页面母版

- 后台 `CRUD / 树表 / 日志` 页面统一取消大页头。
- 页面结构统一为：
  - 筛选操作区
  - 轻量列表头
  - 数据区
- 页面内局部 `刷新` 按钮默认取消。
- `新增` 等主操作下沉到筛选操作区。
- `参数设置 / 表单构建 / 代码生成编辑页 / 服务监控 / 首页类` 保留压缩版轻标题条。
- 路由级标准列表页统一支持列个性化配置：
  - 显示/隐藏
  - 拖拽排序
  - 列宽拖拽
  - 固定列
  - 自动记住
- 列个性化配置第一版只覆盖标准 `ant-design-vue Table` 路由页，不覆盖编辑态子表格、代码生成字段表、`vxe-table`、组件内嵌表格。
- 列配置按“当前用户 + 当前路由 + 当前表格”隔离，默认存浏览器本地。
- `操作` 列属于系统列：
  - 始终显示
  - 始终固定右侧
  - 始终位于最右
  - 允许调整列宽

## 日志与审计

- 操作日志和登录日志写数据库是否启用，由本地配置 `logger.enableddb` 控制。
- 关键系统操作统一复用 `sys_opera_log`，不另建审计表。
- 审计分类采用稳定枚举，不允许页面自行发散命名。

## 开发工具

- `系统接口` 使用现有接口管理页承接。
- `表单构建` 第一版继续使用后端静态页 + Vue bridge。
- `代码生成` 当前重点是“配置可维护 + 预览可验证”。
- `生成到项目` 和 `生成菜单/API` 仍然隐藏，后续单独作为二阶段能力。

## 用户状态协议

- 用户可登录状态统一使用 `status = 2`。
- `status = 1` 表示停用/关闭。
- 前后端均以这套协议为准，不能再混用 `0/1` 语义。

## 前端用户字段展示口径

- 前端界面中的 `username` 统一显示为 `登录账号`。
- 前端界面中的 `realName` / `nickName` 统一显示为 `姓名`。
- 输入提示统一使用 `请输入登录账号`、`请输入姓名`。
- 这组约定只影响前端文案与视觉展示；后端字段名、接口字段名、数据库列名保持原样，不因文案调整而重命名。

## 数据库初始化与迁移

- 左侧导航是否可见，先看数据库初始化是否完整，不要先用前端 fallback 掩盖空库问题。
- `sys_menu` 仍然是菜单真相源；本地库如果 `sys_menu` 为空，优先修数据库初始化链路，而不是改前端渲染逻辑。
- 本地开发统一使用 `go-admin/config/settings.dev.yml` 作为当前联调配置。
- 后端启动统一使用 `./go-admin server -c config/settings.dev.yml`。
- 数据库迁移统一使用 `./go-admin migrate -c config/settings.dev.yml`。
- 当前这套迁移体系是“迁移代码编进 `./go-admin` 二进制”模式；新增 migration 后，必须重新编译再执行迁移。
- 给 `sys_user` 这类现有表新增业务字段时，不能只改 `app/admin/models`；必须补正式 migration，确保存量库和新库都能补齐字段。

## 平台能力层第一期

- 第一批平台能力优先补“审批流 MVP”，而不是先做附件中心。
- 审批流 MVP 应与“模块注册 / 模块入口配置最小版”一起启动。
- Step 1 已按最小增量方式落地到 `go-admin/app/platform`：
  - `module_registry`
  - `wf_definition`
  - `wf_definition_node`
  - `wf_instance`
  - `wf_task`
  - `wf_action_log`
  - `wf_business_binding`
- 第一批正式顺序为：
  - 审批流 MVP + 模块注册 / 模块入口配置
  - 我的待办 / 我发起的 + 统一业务操作日志最小规范
  - 附件中心最小版
  - 财务预算管控模块作为首个接入样板
- 当前已提交代码可直接证明：
  - Step 1 已进入已提交主线
  - Step 2 中“我的待办 / 我发起的 / 流程实例详情抽屉”的最小验收层已进入已提交主线
  - `统一业务操作日志最小规范` 仍未正式闭合
  - Step 3 附件中心最小版已进入已提交主线
  - finance 当前只有样板注册进入已提交主线，业务模块本身仍待收口
- 第一批不做：
  - 会签/或签
  - 条件分支
  - 抄送规则
  - 流程设计器
  - 消息中心
  - 数据权限框架
- 为了先验证审批流主链路，Step 2 先落“前端最小验收层”：
  - `我的待办`
  - `我发起的`
  - 流程实例详情抽屉
  - 页面内最小动作：通过 / 驳回 / 撤回
- 支撑 Step 2 的后端最小列表接口为：
  - `GET /api/v1/platform/workflow/tasks/todo`
  - `GET /api/v1/platform/workflow/instances/started`
- Step 3 附件中心最小版继续挂在 `app/platform`，不塞回 `admin`：
  - 使用 `att_file` 作为附件元数据表
  - 最小接口仅包含上传 / 列表 / 受控下载 / 删除
  - 第一版附件入口先嵌入流程实例详情抽屉，不单独做附件中心页面

## 当前主线切换

- clone 基线本身已经包含同事融合进来的业务代码，不把当前仓库描述成“已删掉业务代码后回到纯脚手架”。
- 当前仓库的后端业务真相源以 clone 基线附近为准；本地叠加的业务后端实验层已基本收口。
- finance 本地扩展已清理，不再作为当前轮次主线继续推进。
- feishu / kingdee / biz_action_log / `sys_user` 扩展类本地后端实验也按同一口径收口。
- 当前主线转向：
  - 前端原型补全
  - 页面与交互完善
  - 脚手架增强
  - 文档收口
- 当前轮次默认不再随意扩写本地后端业务线；如果后续确实要重开，必须先单独对齐真相源边界，再决定是否进入主线。

## 待办与通知中心边界

- `我的待办` 是平台级任务中心，负责承载“需要我处理”的事项。
- 右上角提示图标定义为未来的通知中心入口，负责承载“需要我知道”的事项。
- 待办与通知可以同源，但不能混成同一套列表，也不能共用同一个数量口径。
- `未读消息数` 不等于 `待办数`。
- 消息已读不等于任务已处理。
- 在通知中心未正式建设前，右上角提示图标不承载真实待办数据。
- 当前代码策略为：右上角提示图标先保留占位，不接审批待办真相源。

## 头像下拉与个人中心

- 右上角头像下拉定义为真实用户入口，不再展示框架演示菜单。
- 当前下拉菜单仅保留：
  - `个人中心`
  - `问题&帮助`（占位）
  - `锁屏`
  - `退出登录`
- `文档`、`GitHub` 等演示入口不再出现在头像下拉中。
- `个人中心` 必须稳定打开现有 `/profile` 页面；如果用户已在个人中心，应给出明确提示，而不是静默无反馈。
- 默认头像策略定义为：
  - 用户自定义头像优先
  - 无自定义头像时，展示“姓名优先、登录账号兜底的首字符 + 账号级稳定纯色背景”
- 默认背景色当前以前端确定性映射实现，后续再由个人中心承接“自定义头像 / 默认背景色”的设置能力。
- `backend accessMode` 下，所有不依赖后端菜单的隐藏前端页（如个人中心）必须走静态私有路由，不得挂在 backend 动态菜单路由源中。
- 个人中心的“基本设置”当前定义为只读资料面板，不再复用演示表单。
- 角色、部门、岗位和权限属于平台底座层授权信息，不允许在个人中心中编辑；若需要展示，只能只读展示真实角色信息。
- 个人中心当前允许的最小自维护范围只有：
  - 修改密码
  - 个人简介
- 修改密码必须走真实后端接口 `PUT /api/v1/user/pwd/set`，不允许再保留“前端提示成功但未实际提交”的演示逻辑。
- 密码哈希职责统一由后端 service 层显式生成单次 bcrypt 哈希，不允许在 API 层和 model hook 中重复处理。
- `sys_user.introduction` 作为第一版个人简介持久化字段，不额外新建 profile 扩展表。
- `GET /api/v1/getinfo` 与 `PUT /api/v1/user/profile` 统一以真实 `introduction` 字段为准，不再返回硬编码简介文本。

## 头像系统

- 默认头像必须保持“前端零后端依赖”：
  - 无自定义图片时，前端按“姓名优先、登录账号兜底的首字符 + 稳定纯色背景”实时生成。
  - 后端不再给 `getinfo` 返回外部默认 GIF。
- 用户头像配置方案继续挂在 `sys_user`，不额外新建 profile 扩展表。
- 但头像相关 migration、通用组件和部分页面实现当前仍包含本地未提交状态；在正式 migration 进入已提交链前，不能把这条线写成“仓库主线已完成”。
- 头像模式规则固定为：
  - `avatar_type = image` 且 `avatar` 非空时，展示图片头像。
  - `avatar_type = letter` 时，强制展示字母头像，即使库里还保留旧图片路径。
  - `avatar_type` 为空但 `avatar` 非空时，按旧数据兼容为图片头像。
- 个人中心是头像配置唯一正式入口：
  - 允许切换字母头像/图片头像
  - 允许设置字母头像背景色
  - 允许上传图片头像
- 本地开发环境头像上传统一落到：
  - `go-admin/static/upload/avatar/{userId}/`
- 首版上传链路优先保证：
  - 格式校验
  - 2MB 限制
  - 前端裁剪压缩
  - 全局状态即时刷新
- 字母头像的文字大小占比统一收口到底层 `VbenAvatar`；业务页面不得再单独覆写字号形成新的展示口径。
- 原图/缩略图双版本、OSS 存储属于下一阶段增强，不在本轮首版范围内。
