# 平台能力层第一期最小闭环

## 目的

这份文档定义“平台能力层第一期”的最小落地范围，用于在不做大而全平台的前提下，先补出足以支撑首个真实业务模块接入的公共能力。

当前已确定：

- 首个真实业务模块将以“财务预算管控模块”为样板
- 当前底座层已基本够用
- 第一批必须先补的平台能力，不应再硬写进业务模块内部

## 第一批 P0 能力

### 1. 模块注册 / 模块入口配置最小版

作用：

- 定义一个业务模块如何进入平台
- 统一模块的路由前缀、根菜单和权限前缀约定

第一期最小要求：

- 为模块定义唯一 `module_key`
- 为模块定义唯一 `route_base`
- 为模块定义唯一根菜单入口
- 为模块定义建议权限前缀

第一期不做：

- 模块市场
- 动态安装/卸载
- 模块热启停
- 模块配置中心 UI

### 2. 审批流 MVP

作用：

- 提供首个跨业务复用的平台流程能力

第一期最小要求：

- 顺序审批
- 指定用户 / 指定角色审批
- 提交 / 通过 / 驳回 / 撤回
- 流程定义
- 节点定义
- 流程实例
- 审批任务
- 动作记录
- 状态回写

第一期不做：

- 会签/或签
- 条件分支
- 抄送规则
- 流程设计器
- 多版本编排

### 3. 我的待办 / 我发起的

作用：

- 让审批流具备用户可见、可操作的闭环入口

第一期最小要求：

- 我的待办
- 我发起的
- 审批详情
- 可执行通过 / 驳回 / 撤回
- 可查看流程状态和审批记录

第一期不做：

- 我的已办单独中心
- 催办
- 抄送我
- 批量审批

### 4. 附件中心最小版

作用：

- 让业务附件不再散落在业务表和页面中

第一期最小要求：

- 附件元数据表
- 业务单据绑定
- 上传
- 附件列表
- 受控下载

第一期不做：

- 统一预览中心
- 版本管理
- 分片上传
- 全局附件搜索

### 5. 统一业务操作日志最小规范

作用：

- 把业务动作从系统管理审计中区分出来
- 为审批和首个业务模块提供统一操作记录

第一期最小要求：

- 独立于 `sys_opera_log` 的公共业务操作日志模型
- 审批动作和业务关键动作统一写入
- 业务详情或审批详情可查看业务操作记录

第一期不做：

- 全局业务日志中心
- 审计报表
- 差异分析

## 公共表与业务表边界

平台公共表建议最小集：

- `module_registry` 或等价模块注册表
- `wf_definition`
- `wf_definition_node`
- `wf_instance`
- `wf_task`
- `wf_action_log`
- `att_file`
- `biz_action_log`

财务预算业务表只负责：

- 预算业务字段
- 预算业务状态
- 预算业务单号
- 预算业务摘要
- 可选的 `workflow_instance_id`

不要直接写死到预算业务表的内容：

- 审批节点配置
- 审批任务
- 审批动作时间线
- 附件文件路径集合
- 待办状态
- 业务操作日志明细

## 当前正式实施顺序

### Step 1

- 审批流 MVP 核心后端
- 模块注册 / 模块入口配置最小版

当前已完成：

- 工作流核心数据模型与迁移
- 模块注册数据模型与迁移
- 流程定义 / 节点定义管理接口
- 发起审批 / 通过 / 驳回 / 撤回接口
- 最小业务状态回写绑定
- `finance-budget` 默认模块注册种子数据

### Step 2

- 我的待办 / 我发起的
- 统一业务操作日志最小规范

当前已完成（前端最小验收层）：

- `我的待办` 页面
- `我发起的` 页面

- 页面内最小审批动作：通过 / 驳回 / 撤回

  - `GET /api/v1/platform/workflow/tasks/todo`
  - `GET /api/v1/platform/workflow/instances/started`

### Step 3

- 附件中心最小版

当前已完成：

- 附件元数据表 `att_file`
- 平台附件后端目录与迁移模型
- 附件上传、附件列表、受控下载、删除接口
- 关联到流程实例详情抽屉的最小附件区块
- Step 3 live 联调与 Playwright 附件 smoke 通过

### Step 4

- 财务预算管控模块作为首个样板接入

## 为什么不是“先附件”

附件虽然也是 P0，但它不是首个业务模块接入时最核心的阻塞项。

对财务预算模块来说，最先必须跑通的是：

1. 提交审批
2. 待办处理
3. 状态回写

因此第一批顺序采用：

1. 先审批
2. 再待办
3. 再附件
4. 最后接业务样板

## 开工判断

当前第一期最小闭环已经足够明确，可以直接进入拆任务阶段。

首个开工能力：

- 审批流 MVP

但应与“模块注册 / 模块入口配置最小版”一起启动，以保证首个业务模块接入方式从一开始就是统一的。
